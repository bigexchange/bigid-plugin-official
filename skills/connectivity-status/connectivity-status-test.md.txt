Tests: connectivity-status
________________


Functional tests
Test 1: Standard full-environment health check
Input: "What is the connectivity status of my BigID?"


Expected output:


* Calls post_v1_ds_connections (Data Sources API) to retrieve all data sources
* Calls get_v1_data_catalog_count (Data Catalog) per source with total_pii_count > 0 filter — NOT the DS connections API fields
* Calls get_v1_data_catalog_count with open_access = true filter per source
* Produces a self-contained HTML file at /mnt/user-data/outputs/connectivity-status.html
* Calls present_files with the output path
* HTML contains 6 metric summary cards: Total, Healthy, Issues/Stale, Failed/Never scanned, Sources with PII, Open access + PII
* HTML table contains all 10 columns: Name · Type · Status · Last scanned · Days ago · Conn · Scanned · PII (objects) · Access · Next step
* Each row has a color-coded Status badge (green / orange / red / gray)
* PII column shows "Y · N objects" badge (coral) or "N" (green) or "—" (gray) — never a raw number from the DS API
* Scanned column shows ✅ when last_full_scan_at is present, ❌ when absent
* Access column shows "Restricted" or "Open" badge — never blank for scanned sources
* Next step column is populated for every row (≤8 words, concise)
* Rows sorted: issues/failed first, then healthy-with-PII by object count desc, then healthy-no-PII, then draft
* Post-render prose summary (3–5 sentences) written after the file link
* No raw API field names or JSON exposed to the user


________________


Test 2: Environment with a mix of status buckets
Input: "Show me my connector health" (against an environment that has at least one stale source, one with a failed last scan, one never scanned, and several healthy)


Expected output:


* Stale sources (last_full_scan_at > 180 days ago) are classified as "Stale" with orange badge
* Source with scan_is_success === false AND has last_full_scan_at is classified as "Last scan failed" (orange), not "Never scanned"
* Source with no last_full_scan_at is classified as "Never scanned" — Scanned column shows ❌
* Stale source with PII = Y gets next step "Re-scan to verify PII scope — data may have changed"
* Stale source with PII = Y AND Access = Open gets next step prefixed with ⚠️
* Summary prose calls out the stale and failed sources by name


________________


Test 3: PII confirmed via Catalog API, not DS API
Input: "Are all my connectors working?" (against an environment where DS connections API returns no findings fields but Catalog API returns count > 0 for several sources)


Expected output:


* PII column shows "Y" with object count for sources where Catalog API count > 0
* PII column shows "N" for sources where Catalog API count = 0 and scan completed
* PII column shows "—" only for sources that have never been scanned
* No row shows PII derived from findings_count, total_findings, or numberOfFindings fields on the DS object
* Summary prose highlights sources with confirmed PII


________________


Test 4: Open access PII exposure risk
Input: "Which data sources have issues?" (against an environment where at least one source has open_access = true in the catalog AND PII findings > 0)


Expected output:


* Access column shows "Open" badge (red) for that source
* Next step cell shows ⚠️ prefix: "Review access controls — sensitive data in open source"
* "Open access + PII" metric card shows a count > 0 (displayed in red)
* Summary prose explicitly calls out the open-access PII source as the highest-priority remediation item
* No other source incorrectly shows "Open" access


________________


Test 5: Large environment (>200 data sources)
Input: "Show me data source connectivity" (against an environment with 250+ data sources)


Expected output:


* Skill paginates post_v1_ds_connections — makes at least 2 calls (limit 200, then remainder)
* All sources are included in the HTML table — none silently dropped
* Metric card "Total sources" reflects the full paginated count
* Per-source Catalog API calls are run for all sources (parallelised)
* HTML file is written and presented correctly despite large row count


________________


Test 6: Alternate trigger phrasings
Input variants to verify skill triggers correctly:


* "Which connectors haven't been scanned recently?"
* "Are all my connectors working?"
* "Which data sources have issues?"
* "Show me my connector health"


Expected output:


* Skill triggers on all four phrasings without requiring rephrasing
* Output structure is identical across all trigger variants
* No variant causes the skill to skip the Catalog API PII enrichment step


________________


Edge cases
* Empty environment: DS connections API returns an empty list → skill states "no data sources found" and asks user to confirm the environment; does not attempt Catalog API calls or produce an HTML file
* All sources healthy, no PII anywhere: Catalog API returns count = 0 for all sources → all PII cells show "N" (green), "Open access + PII" card shows 0 (green), summary prose confirms clean state
* Source with scan_is_success field absent: Treated as unknown — not classified as "Last scan failed"; classified by last_full_scan_at staleness instead
* Catalog API call fails for one source: That source's PII shows "—" (unknown); skill does not abort; other sources are classified correctly
* Source with last_full_scan_at present but scan_is_success === false: Classified as "Last scan failed" (priority 5), Scanned = ✅ (scan ran, even though it failed)
* Source with enabled !== "yes": Classified as "Draft" (priority 1) regardless of any scan or connection history
* Scanner group named "public-dmz": Access classified as "Open" via the name heuristic (contains "public" or "dmz")
* Vague input — "status?": Skill should still trigger and run the full workflow, treating it as a connector health query
